# NUVIO Product Guide

Last updated: 2026-05-25

이 문서는 누비오 화면, 라우팅, 문구, 데이터 이름을 수정할 때 우선 적용하는 제품 가이드입니다. 새 기능을 만들거나 기존 화면을 정리할 때 이 문서를 먼저 확인합니다.

## Product Direction

누비오는 지역 여행 프로그램을 찾는 누비어와 프로그램을 운영하는 호스트를 연결하는 서비스입니다. 단순 여행 검색 사이트가 아니라 신청, 선정, 안내, 후기, 저장, 회원정보 관리까지 이어지는 흐름을 한 제품 안에서 처리합니다.

공개 서비스의 첫 사용자는 누비어입니다. 누비어는 회원가입만 하면 프로그램 신청, 저장, 후기, 메시지, 회원정보 관리를 할 수 있어야 합니다.

호스트센터는 로컬페이지와 프로그램을 운영하는 사람을 위한 공간입니다. 호스트가 처음 들어왔는지 여부보다 해당 계정에 로컬페이지가 있는지 여부가 더 중요한 분기 기준입니다.

## User Roles

### 누비어

- 기존의 참가자, 참여자를 대체하는 사용자 명칭입니다.
- 프로그램을 찾고, 신청하고, 내 여행 프로그램을 확인하고, 후기를 작성합니다.
- 개인 대시보드의 대표 경로는 `/mypage`입니다.
- `/me`는 호환 경로로 둘 수 있지만, 새 화면과 문서에서는 `/mypage`를 우선 사용합니다.

### 호스트

- 기존의 운영자, 운영기관 담당자를 대체하는 명칭입니다.
- 로컬페이지, 폴더, 프로그램, 신청자, 메시지, 후기, 마감 자료를 관리합니다.
- 호스트 기능의 대표 경로는 `/host`입니다.

### 관리자

- 플랫폼 전체 운영을 위한 내부 권한입니다.
- 관리자 화면에서는 플랫폼 운영 주체를 가리킬 때만 관리자라는 말을 씁니다.
- 일반 호스트를 관리자 또는 운영자로 부르지 않습니다.

## Core Routes

| Area | Canonical Route | Notes |
| --- | --- | --- |
| 마이페이지 | `/mypage` | 누비어 개인 대시보드 |
| 이전 내 페이지 | `/me` | 호환 경로. 새 문서에서는 `/mypage` 사용 |
| 호스트센터 | `/host` | 로컬페이지 유무에 따라 화면 분기 |
| 프로그램 탐색 | `/programs`, `/search` | 신청 가능한 여행 프로그램 탐색 |
| 프로그램 상세 | `/programs/[id]` | 신청, 저장, 후기 진입 |
| 로컬페이지 | `/[villageSlug]` | 내부 모델명은 `village`가 남아도 사용자 용어는 로컬페이지 |

## Host Center Flow

호스트센터의 첫 분기는 "처음 방문했는가"가 아니라 "계정에 로컬페이지가 있는가"입니다.

### 로컬페이지가 없는 경우

`/host`에서 바로 로컬페이지 만들기 화면을 보여줍니다.

필수 입력은 간단하게 유지합니다.

- 대표 이미지
- 위치
- 로컬페이지 이름
- 짧은 소개글

이 화면에서는 "로컬홈"이라는 말을 쓰지 않습니다. 항상 "로컬페이지"라고 부릅니다.

### 로컬페이지가 있는 경우

`/host`는 호스트센터 대시보드를 보여줍니다.

기본 구성은 다음 순서입니다.

1. 사이드탭
2. 폴더 목록 영역
3. 첫 번째 카드: 새 폴더 만들기
4. 프로그램 영역
5. 첫 번째 카드: 새 프로그램 만들기
6. 기존 프로그램 목록

기존에 프로젝트라고 불렀던 개념은 사용자 화면에서 폴더라고 부릅니다. 내부 라우트나 DB 호환성 때문에 `projectId`, `/host/projects`, `report_projects`가 남을 수 있지만, 화면 문구는 폴더로 맞춥니다.

폴더를 클릭하면 해당 폴더 안의 프로그램 목록을 보여주는 흐름을 기본으로 봅니다.

새 폴더 만들기는 `/host/projects/new` 같은 별도 주소를 주요 UX로 쓰지 않고, `/host` 안에서 폴더 이름을 입력하는 팝업 형태를 우선합니다. 기존 주소는 호환용으로만 유지할 수 있습니다.

## MyPage Flow

마이페이지는 누비어의 개인 대시보드입니다.

기본 경로는 `/mypage`이고, 화면 파일명과 컴포넌트 이름도 가능하면 `mypage` 계열로 통일합니다.

주요 하위 영역은 다음과 같습니다.

- 내 여행 프로그램
- 후기
- 저장
- 메시지함
- 회원정보
- 포인트
- 쿠폰함
- 설정
- 고객센터
- 로그아웃

### 회원정보

회원정보는 기본 보기와 수정 모드를 구분합니다.

기본 보기에서는 입력 필드를 노출하지 않고 저장된 정보를 읽기 형태로 보여줍니다.

수정 모드에서는 입력 필드를 노출하고 저장하기 버튼을 제공합니다.

소셜 로그인 계정은 일반 로그인 아이디와 비밀번호가 없을 수 있습니다. 이 경우 아이디/비밀번호 필드를 억지로 요구하지 않고, 연결된 이메일 또는 소셜 계정 상태를 보여주는 방향을 우선합니다.

이메일은 아이디 부분과 도메인 부분을 나누고, 도메인은 드롭다운과 직접 입력을 함께 지원합니다.

생년월일은 드롭다운을 사용합니다. 기본 시작 연도는 현재 연도 기준 성인이 되는 해입니다. 2026년 기준 기본 연도는 2007년입니다.

주소 검색은 팝업 또는 외부 주소 검색 위젯을 통해 입력하게 합니다. 외부 API가 차단되거나 스크립트 로딩이 실패하는 경우, 사용자가 직접 입력할 수 있는 fallback을 제공해야 합니다.

## Header And Profile

사이트 헤더는 로그인 전후 상태가 달라야 합니다.

로그인 전에는 로그인 버튼을 보여줍니다.

로그인 후에는 로그인 버튼을 숨기고 알림 아이콘과 프로필 버튼을 보여줍니다. 프로필 버튼을 누르면 작은 계정 팝업을 표시합니다.

프로필 팝업에는 마이페이지 이동, 주요 지표, 로그아웃, 설정 진입을 담을 수 있습니다. 1440px 디자인 기준 팝업 너비가 212px이었다면, 실제 구현에서는 화면 비율에 맞게 자연스럽게 확장하되 과도하게 커지지 않도록 합니다.

## Allowed Compatibility Terms

아래 표현은 사용자에게 직접 노출하지 않는 내부 호환성 이름으로 남길 수 있습니다.

| Internal Term | Reason |
| --- | --- |
| `bookmark`, `bookmarks` | 기존 라우트와 저장 상태 API 호환성 |
| `saved_programs` | DB 테이블명 |
| `village`, `village_page_sections` | 기존 로컬페이지 모델과 CMS 테이블명 |
| `projectId`, `/host/projects`, `report_projects` | 호스트 폴더 기능의 기존 라우트/DB 호환성 |
| 참가자, 참여자 | 공식 공지 또는 원문 후기 인용문 |
| 운영자 | 법무 문서에서 서비스 운영 주체를 뜻하는 경우 |

## Copy Style

누비오 문구는 부드러운 해요체를 기본으로 합니다.

너무 딱딱한 행정 문구보다 사용자가 다음 행동을 쉽게 이해하는 문장을 씁니다.

버튼은 명사보다 동사를 우선합니다.

- 좋은 예: 신청하기, 저장하기, 후기 작성하기, 로컬페이지 만들기
- 피할 예: 신청, 저장, 작성, 생성

검색 관련 문구는 다음 기준을 우선합니다.

- 헤더 검색 placeholder: `새로운 여행을 떠나볼까요?`
- 검색 모달 제목: `어떤 여행을 떠나고 싶나요?`
- 검색 모달 placeholder: `새로운 여행 프로그램을 검색해보세요`

검색 모달은 인기 프로그램, 테마, 지역, 일정 중심으로 구성합니다. 날짜와 인원 선택을 먼저 강요하는 구조는 우선하지 않습니다.

## Layout Rules

Figma 기준 화면은 1440px로 작업했더라도 실제 구현의 최대 폭은 1920px 기준으로 확장 가능해야 합니다.

고정 `max-width: 1080px`처럼 1440 디자인의 내부 폭을 그대로 박아두지 않습니다. 더 넓은 화면에서는 비율에 맞게 자연스럽게 늘어나야 합니다.

피그마 기반 화면의 실제 구현 방법론은 `docs/figma-to-web-implementation-guide.md`를 기준으로 합니다. 폰트 크기, 아이콘 크기, 여백, 선, 라운드도 1440에서 1920까지 같은 비율로 확장되어야 합니다.

히어로와 헤더 사이 간격은 과도하게 넓히지 않습니다. 실제 화면에서 시각적으로 붕 뜨면 절반 수준으로 줄이는 것을 우선 검토합니다.

반복 카드, 폴더, 프로그램 목록은 화면 폭에 맞춰 열 개수가 자연스럽게 변해야 합니다.

## Implementation Checklist

새 화면 또는 리팩터링을 할 때 아래를 확인합니다.

1. 사용자 노출 문구에 금지 용어가 남아 있지 않은가?
2. 내부 호환성 이름과 사용자 문구를 혼동하지 않았는가?
3. `/mypage`, `/host`의 대표 흐름을 깨지 않았는가?
4. 로그인 전후 헤더 상태가 분리되어 있는가?
5. 1440px뿐 아니라 1920px까지 자연스럽게 확장되는가?
6. 모바일에서 텍스트와 버튼이 겹치지 않는가?
7. 기존 DB 데이터에 남은 옛 용어가 필요하면 마이그레이션으로 보정했는가?
8. `rg` 검색으로 핵심 금지 용어를 확인했는가?
9. `npm run lint`, `npx tsc --noEmit`, `npm run build`를 통과했는가?

## Search Commands

용어 정리 후에는 아래 검색을 우선 실행합니다.

```bash
rg -n "로컬홈|로컬 홈|local-home|Local Home|마을 페이지|마을 홈페이지|로컬 채널|운영 채널|마을 채널|북마크|매거진|메세지|신청 내역|예약번호|어디로 떠날까요|참여자 리뷰|참가자 리뷰|로컬프로젝트|운영기관|운영자" src --glob "!src/lib/legal-documents.ts"
```

```bash
rg -n "참여자|참가자" src/components src/app src/lib
```

위 검색에서 남아도 되는 결과는 크롤링 키워드, 원문 후기 인용문, 법무 문서처럼 문맥상 필요한 경우뿐입니다.
