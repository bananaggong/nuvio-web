# NUVIO 기획 정리 및 3회 반복 평가

작성일: 2026-05-06

## 1. 현재 RSS 공고 오탐 원인

이전 구현의 `/api/programs` 기준 공개 목록은 총 13건이었고, 이 중 12건은 RSS에서 자동 수집한 외부 후보, 1건은 Supabase에 직접 게시된 프로그램이었습니다. 문제는 외부 후보 12건 대부분이 “여행자 대상 지역 체류/워케이션/한달살기 모집”이 아니라 콘텐츠 지원사업, 연구개발 선정 결과, 위탁용역 선정 결과, 교육생 모집 등이라는 점입니다.

2026-05-06 개선 이후 공개 목록은 미검수 RSS 후보를 기본 노출하지 않고, DB 게시 프로그램과 NUVIO가 직접 구성한 curated 프로그램을 노출합니다. RSS 후보는 프로그램 발굴/검수 큐로 다루는 것이 맞습니다.

### 노출된 외부 후보의 실제 성격

이전 기준에서 “NUVIO 공개 홈에 바로 보여도 되는 외부 후보”는 12건 중 0건에 가깝습니다. 일부는 문화/관광 정책과 멀리 연결될 수 있지만, 참가자가 신청할 체류 프로그램으로 보기 어렵습니다.

### 코드 기준 원인

1. 소스 자체가 너무 넓습니다.

   `src/lib/announcement-sources.ts`의 `kocca-notice`는 한국콘텐츠진흥원 전체 공지 RSS입니다. 키워드가 `지역`, `관광`, `콘텐츠`, `공모`, `모집`, `지원`, `체험`이고 `minimumKeywordMatches`가 2라서 콘텐츠 지원사업 공지가 쉽게 통과합니다.

2. 키워드가 너무 일반적입니다.

   `지역`, `지원`, `공모`, `선정`, `콘텐츠` 같은 단어는 NUVIO의 핵심인 “여행자/참가자 대상 지역 체류”를 보장하지 않습니다. 하지만 현재 로직에서는 이런 단어들이 관련성 점수로 더해집니다.

3. 점수 기준이 낮고 공개 목록에서 하드코딩되어 있습니다.

   `src/lib/crawled-programs.ts`는 공개 후보 생성 시 `minimumScore: 2`를 직접 넘깁니다. 환경변수 `PROGRAM_LEAD_MIN_SCORE`보다 낮은 기준이 공개 화면에 적용될 수 있습니다.

4. “결과 안내”와 “선정평가 결과”가 모집 신호처럼 처리됩니다.

   `src/lib/program-leads.ts`의 긍정 규칙에 `선정`, `공모`, `지원`이 포함되어 있습니다. 그래서 `선정 결과`, `접수결과`, `최종 결과 안내`가 새로 신청 가능한 프로그램처럼 점수를 얻습니다.

5. 공개 홈에 관리자 검수 게이트가 없습니다.

   현재 구조는 외부 공고 후보를 바로 공개 프로그램 카드로 변환합니다. 후보 수집과 공개 게시가 분리되어 있지 않기 때문에 오탐이 곧바로 사용자에게 보입니다.

6. 모든 외부 후보에 `benefit` 카테고리가 붙습니다.

   `src/lib/crawled-programs.ts`에서 외부 후보의 categories에 항상 `benefit`을 추가합니다. 실제 지원 혜택이 확인되지 않은 공고도 지원혜택처럼 필터에 잡힙니다.

7. 본문 상세 파싱이 부족합니다.

   RSS title/description만으로 판단하고 있어 신청 대상, 모집 기간, 참가자 유형, 숙박/체류 여부를 구조화해서 확인하지 못합니다.

## 2. NUVIO 기획 요약

### 제품 정의

NUVIO는 지역 체류 프로그램을 찾는 참가자와 프로그램을 운영하는 마을/기관을 연결하는 로컬 체류 운영 플랫폼입니다. 단순 홍보 사이트가 아니라 신청, 선정, 입금/결제, 공지, 기수 커뮤니티, 후기, 만족도 조사, 보고서까지 이어지는 운영 인프라가 핵심입니다.

### 핵심 문제

- 구글폼, 개인 문자, 단체 카카오톡방, 엑셀 보고서가 흩어져 운영됨
- 기수별 공지방/후기방/신청자 정보가 관리되지 않음
- 신청자가 본인이 신청한 정보와 이후 안내를 다시 찾기 어려움
- 운영자는 선정, 입금, 톡방 초대, 만족도 조사, 후기 수집을 반복 수작업으로 처리함
- 마을별 홍보 페이지와 신청/운영 시스템이 분리되어 있음
- 중복 참여자, 신원 확인, 입금 여부, 보고서 제출용 데이터 관리가 어렵다

### 사용자

| 사용자 | 목표 |
| --- | --- |
| 참가자 | 지역 체류 프로그램 탐색, 신청, 선정 안내 확인, 결제/입금, 공지 확인, 후기 작성 |
| 마을/운영자 | 프로그램 홍보, 신청자 관리, 선정/입금 확인, 기수별 공지, 설문/후기 수집, 보고서 작성 |
| NUVIO 관리자 | 공고 소스 관리, 프로그램 품질 관리, 파트너/마을 운영 현황 확인 |

### Layer 1: 공개 탐색/신청 레이어

- 지역 체류, 워케이션, 반값여행, 한달살기 프로그램 목록
- 프로그램 상세와 공식 원문/신청 안내
- 마을별 홈페이지: `/villages/[slug]`, `/[villageSlug]`
- 마을별 프로그램 상세: `/[villageSlug]/[programSlug]`
- 후기, 공지, 로그인, 마이페이지
- 외부 공고 후보 수집은 가능하지만 공개 노출은 검수된 항목만 해야 함

### Layer 2: 호스트 운영 SaaS 레이어

- 마을 홈페이지 스튜디오
- 프로그램 스튜디오
- 신청서 빌더
- 신청자 CRM과 상태 관리
- 선정/대기/탈락/입금/참여/완료 상태 전환
- 메시지 템플릿과 자동 발송 대기열
- 기수별 공지/커뮤니티 구조
- 사전 만족도, 기대치 설문, 활동 후기, 리뷰 수집
- 보고서용 엑셀/PDF export
- 결제/입금 확인 연동
- 중복 참여자와 신원 확인 관리

### URL 전략

초기에는 커스텀 도메인과 서브도메인을 쓰지 않고 경로형으로 간다.

- 마을 목록: `/villages`
- 마을 홈: `/villages/[slug]`, `/[villageSlug]`
- 마을별 프로그램: `/[villageSlug]/[programSlug]`

이 방식은 설정과 운영 비용이 낮고, Codex가 자동으로 라우팅/SEO/권한 작업을 이어가기 쉽다.

## 3. 반복 평가 1회차: 데이터 정확도 관점

### 평가

현재 외부 공고 수집은 “RSS를 가져오는 기능” 자체는 동작하지만, 공개 서비스에 필요한 precision이 낮습니다. 단어 단위 규칙만으로는 콘텐츠 지원사업, 연구개발 결과, 위탁용역 선정 결과를 여행지원금 프로그램과 구분하지 못합니다.

점수: 4 / 10

### 주요 문제

- 공개 홈이 자동 후보를 바로 보여준다.
- `지역`, `지원`, `공모` 같은 넓은 키워드가 과대평가된다.
- `선정 결과`, `최종 결과`, `접수결과` 같은 종료/결과성 공지가 차단되지 않는다.
- KOCCA 전체 공지는 공개 프로그램 소스로 쓰기에 너무 넓다.

### 개선점

1. 공개 홈은 승인된 프로그램만 노출한다.
2. 외부 RSS 후보는 `/admin` 또는 호스트 검수 큐에만 보여준다.
3. 공개 후보 조건에 필수 앵커를 둔다.
   - 여행/체류 앵커: `한달살기`, `살아보기`, `워케이션`, `숙박`, `체류`, `여행자`, `관광객`, `지역상품권`, `팸투어`, `로컬체험`
   - 모집 앵커: `참가자 모집`, `신청 접수`, `체험단 모집`, `서포터즈 모집`
4. 제외 앵커를 추가한다.
   - `선정 결과`, `평가 결과`, `최종 결과`, `접수결과`, `위탁용역`, `연구개발`, `채용`, `입찰`, `행정처분`, `후보자 추천`
5. `crawled-programs.ts`의 hardcoded `minimumScore: 2`를 제거하고 환경변수 또는 별도 공개 기준을 사용한다.

## 4. 반복 평가 2회차: 제품/운영 적합성 관점

### 평가

1회차 개선만 하면 오탐은 줄어들지만, NUVIO의 본질은 “공고 검색 사이트”가 아니라 “운영을 끝까지 처리하는 SaaS”입니다. 따라서 외부 공고 수집은 공개 홈의 메인 데이터가 아니라 운영자가 프로그램을 발굴하는 보조 도구여야 합니다.

점수: 6.5 / 10

### 새로 발견한 문제

- “외부 후보”와 “운영 가능한 프로그램”의 상태 구분이 약하다.
- RSS 후보에는 신청 URL, 모집 기간, 참가비, 혜택, 인원, 숙소 정보가 대부분 없다.
- 사용자가 외부 후보를 보고 신청하려 해도 NUVIO 내부 신청/결제/기수 관리로 이어지지 않는다.
- 마을별 홈페이지와 프로그램 운영 SaaS의 가치가 공개 RSS 카드에 묻힌다.

### 개선점

1. 데이터 상태를 명확히 나눈다.
   - `external_candidate`: 외부에서 찾은 후보
   - `verified_opportunity`: 운영자가 검수한 외부 기회
   - `host_program`: NUVIO에서 운영/신청 가능한 프로그램
   - `archived_notice`: 결과/마감/정책 공지
2. 홈의 기본 탭은 `NUVIO 신청 가능`으로 두고, 외부 후보는 별도 탭이나 관리자 영역으로 뺀다.
3. 외부 후보 승인 시 운영자가 최소 필드를 보강하게 한다.
   - 모집 기간
   - 신청 대상
   - 혜택/금액
   - 체류 기간
   - 숙소/장소
   - 신청 링크
   - 담당 기관
4. 프로그램 상세에서 외부 후보는 “신청하기”가 아니라 “검수 전 후보”로 표시한다.
5. 마을 페이지는 프로그램 신청, 공지, 후기, 기수 커뮤니티로 이어지는 운영 중심 화면으로 강화한다.

## 5. 반복 평가 3회차: 사업화/확장성 관점

### 평가

2회차까지는 제품 품질을 올리는 개선입니다. 다만 빠르게 selling 가능한 수준으로 가려면 “좋은 데이터”보다 “운영자가 매일 쓰는 업무 흐름”이 더 중요합니다. 현재 가장 큰 사업적 차별점은 공고 크롤링이 아니라 마을 운영의 반복 업무를 줄이는 것입니다.

점수: 8 / 10이 될 수 있는 방향

### 새로 발견한 문제

- 자동 수집 공고를 많이 보여주는 것은 오히려 신뢰를 해칠 수 있다.
- 초기 고객인 마을/청년마을 운영자에게 필요한 것은 “외부 공고 100개”보다 “내 프로그램 운영이 편해지는 것”이다.
- 입금 확인, 단톡방 안내, 기수별 공지, 후기/만족도 수집, 보고서 export가 실제 구매 이유에 가깝다.
- 참가자 커뮤니티는 개인정보와 방치 문제를 해결하지 못하면 오히려 운영 부담이 된다.

### 개선점

1. MVP 이후 바로 팔 수 있는 패키지를 명확히 한다.
   - 마을 홈 1개
   - 프로그램 3개
   - 신청서 3개
   - 신청자 상태 관리
   - 입금 확인 컬럼
   - 기수별 공지 페이지
   - 후기/만족도 수집
   - 보고서 CSV/XLSX export
2. 결제/입금 확인은 단계적으로 간다.
   - 1단계: 운영자가 수동 체크
   - 2단계: 입금자명/금액/시간을 CSV 업로드로 매칭
   - 3단계: PG 또는 가상계좌 API 연동
3. 커뮤니티는 카카오톡방 대체가 아니라 공식 공지/자료 보관소부터 시작한다.
   - 공지용 페이지
   - 기수별 자료
   - 후기방
   - 개인정보 가림
   - 종료 후 소식 구독
4. 외부 공고 수집은 “NUVIO 운영자가 새로운 영업/제휴 후보를 찾는 기능”으로 재배치한다.
5. 첫 유료 고객을 위해 보성형 템플릿을 만든다.
   - 문화체험 4회
   - 살아보기 8회
   - 블렌딩티 등 프로그램 여러 개
   - 기수별 신청/선정/입금/공지/후기/보고서 흐름

## 6. 최종 권장 우선순위

### P0: 즉시 수정

1. 공개 홈에서 외부 RSS 후보를 기본 노출하지 않는다.
2. KOCCA RSS는 public source가 아니라 admin lead source로 낮춘다.
3. `선정 결과`, `평가 결과`, `최종 결과`, `위탁용역`, `연구개발`, `채용`, `입찰` 차단 규칙을 추가한다.
4. 공개 후보에는 여행/체류 필수 앵커를 요구한다.
5. 모든 외부 후보에 `benefit`을 자동 부여하지 않는다.

### P1: 이번 제품 고도화 사이클

1. 외부 후보 검수 큐를 강화한다.
2. 후보 승인 시 프로그램 초안 생성 플로우를 개선한다.
3. 마을 홈에서 신청 가능한 프로그램과 공지/후기를 자연스럽게 연결한다.
4. 신청자 상태 관리에 입금 확인과 기수 배정을 추가한다.
5. 보고서 export에 필요한 필드 매핑을 신청서 빌더와 연결한다.

### P2: 판매 가능한 운영 패키지

1. 보성 청년마을 기준 운영 템플릿
2. 신청서-선정-입금-공지-후기-보고서 end-to-end 데모
3. 관리자 역할/권한 정책
4. PG 또는 입금 매칭 연동
5. Vercel/Supabase 운영 모니터링과 데이터 백업 정책

## 7. 결론

RSS 오탐의 근본 원인은 “실제 크롤링 실패”가 아니라 “수집 후보와 공개 프로그램의 경계가 없고, 키워드 판별이 너무 넓은 것”입니다. 현재 12건은 크롤 자체는 되었지만, NUVIO의 공개 프로그램 기준에는 맞지 않습니다.

다음 개발은 외부 RSS를 더 많이 가져오는 방향이 아니라, 공개 노출 기준을 엄격히 만들고 운영자가 승인한 것만 프로그램으로 전환하는 방향이어야 합니다. 동시에 NUVIO의 판매 포인트는 공고 수집보다 마을 운영자의 반복 업무를 줄이는 Layer 2 운영 흐름에 두는 것이 맞습니다.
